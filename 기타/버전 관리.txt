CVS
-서버에서 버전의 히스토리를 관리
- 각 개발자들이 원하는 내용을 서버에 업데이트하여 동기화
- 중앙 서버에 문제 발생 시, 개발자들의 업무 불가능

DVC
- Distributed Version Control
- 서버 뿐만 아니라, 각 개발자들이 동일한 버전의 히스토리를 가지고 있는 구조
- 서버에 문제 발생 시, 복원이 가능하며 개발자들의 업무가 가능

Git 
	- 로컬 저장소의 버전 관리 시스템
	- 소스 코드나 파일 버전을 관리함
	- 개발자들의 각 작업 컴퓨터에 있는 버전 관리 시스템

Git Hub
	- 클라우드 저장소를 저장하는 형태의 버전 관리 시스템
	- public(공개 저장소), private(비공개 저장소)로 구분할 수 있다.

소스트리 
	- Git을 통한 버전 관리를 할 수 있는 GUI Tool

Working Copy
	- 관리하고 있는 폴더에서 변경사항이 발생하는 파일들이 확인되는 공간
	- 새로 생성한 파일, 기존 파일에서 변경된 파일, 삭제된 파일을 각각 다른 아이콘으로 표시
stage area
	- working copy 영역에 있는 파일 중, 수정이 완료되어 다음 버전에 적용하기 위한 파일들이 올라오는 공간
	- 커밋을 하는 경우, stage area 영역에 있는 파일들의 변경사항만 다음 버전에 적용됨

add
	- workint copy 영역에서, 다음 버전에 적용할 파일들을 stage area에 올리는 행위

commit
	- 하나의 버전을 생성하는 행위
	- commit 시, 해당 버전에 적용되는 내용(메시지)을 작성

- 커밋 이전의 폐기
	- 파일 내용 수정 후, 커밋하기 이전에 '폐기'버튼을 눌러, 변경사항을 복구
	- 파일 내용이 변경 이전의 상태로 돌아감

- 커밋 이후의 폐기(reset)
	- "이 커밋까지 브랜치를 초기화" 버튼을 눌러 진행
	- 선택할 수 있는 옵션 Soft, Mixed, Hard
	- Soft : 선택한 버전으로 되돌리고, 변경사항을 stage area에 위치시킨다
	- Mixed : 선택한 버전으로 되돌리고, 변경사항을 working copy에 위치시킨다
	- Hard : 선택한 버전으로 되돌리고, 파일의 변경사항도 선택한 버전으로 reset

- 커밋 이후 revert (  '*'은 중요한 얘기 )
	- "커밋 되돌리기" 버튼을 눌러 진행
	- 선택한 버전의 변경사항을 되돌리며, 되돌린 상태를 하나의 새로운 버전으로 관리
	- * 가장 최신 버전을 클릭하고 진행활 것 
	- * 가장 최신 버전부터 순차적으로 revert를 진행할 것
	- * 원격 저장소 연동 시, reset은 사용하지 않고, revert만 사용할것(사고 예방)

- push
	- 로컬 저장소의 버전을 원격 저장소에 동기화
	- push 이전에 pull을 진행할 것

- pull
	- 원격 저장소의 버전을 로컬 저장소에 동기화
	
- 충돌 
	- 원격 저장소의 버전을 로컬 저장소에 하나의 버전으로 합치는 과정에서 여러 변경사항이 동시에 발생했을 때
	  git이 스스로 판단할 수 없는 경우 충돌이 발생

- branch(브랜치)
	- 나뭇가지라는 뜻으로. 코드를 복사해서 또 하나의 다른 작업 흐름으로 분리해내는 것
	- 여러 작업흐름을 만들어서 동시에 작업하고 합치는 과정을 통해 개발 속도 향상
	- 개발 중, 문제가 발생하더라도 원래 작업 흐름에는 영향을 주지 않고 해당 작업 흐름에만 문제가 발생하여
	  협업 시, 문제 발생을 줄여줌
	- 저장소 생성 시, 기본적으로 master 브랜치가 생성되는데, master 브랜치에서는 개발하지 말 것
	  별도의 브랜치를 생성하여 해당 브랜치에 개발 진행할 것
	- 브랜치를 생성하는 기준은 매우 다양하다.

-git ignore
	- 버전관리 대상에서 제외할 파일들에 대한 관리 정보
	-workspace 또는 프로젝트 생성 똔는 생성 변경 등의 액션 시, 무수히 많은 폴더 및 몇 파일들이 자동 생성된다.
	그 중에서 버전 관리 대상이 아닌 폴더 또는
